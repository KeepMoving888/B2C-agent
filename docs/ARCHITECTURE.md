# B2C 多 Agent 智能客服 - 架构设计

## 一、业务背景与目标

### 业务场景

- **公司**：跨境 CarPlay 类电商，覆盖欧美、东南亚 12 国 8 种语言
- **渠道**：独立站 + 亚马逊多站点

### 核心痛点

| 痛点      | 量化           | 解决方案            |
| ------- | ------------ | --------------- |
| 客服人力成本高 | 20 人 7×24 排班 | Agent 7×24 自动接待 |
| 旺季响应慢   | 咨询量暴涨 300%   | 自动分流，优先处理复杂问题   |
| 重复咨询占比高 | 70%+         | RAG + 意图路由，自动解答 |
| 信息分散易错  | 多语言/多平台/多物流商 | 统一工具 API + 知识库  |

### 项目目标

搭建全流程智能 Agent，实现 **7×24 多语言咨询接待**，完成 **订单、物流、售后** 等核心场景的 **自动化闭环**，降低人工成本，提升用户体验。

```
                                      │
    ┌─────────────┬─────────────┬─────┴─────┬─────────────┬─────────────┐

    ▼             ▼             ▼           ▼             ▼             ▼
┌────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐  ┌──────────┐
│咨询接待 │  │订单履约  │  │售后处理  │  │合规解答 │  │人工转接  │
│Agent   │  │Agent     │  │Agent     │  │Agent   │  │Agent     │
└────────┘  └──────────┘  └──────────┘  └────────┘  └──────────┘

```

### Agent 职责边界

| Agent    | 职责         | 主要能力         | 工具/RAG              |
| -------- | ---------- | ------------ | ------------------- |
| **咨询接待** | 产品、价格、FAQ  | RAG 知识库检索    | 产品手册、FAQ            |
| **订单履约** | 订单状态、物流追踪  | 订单/物流 API    | 订单系统、物流商 API        |
| **售后处理** | 退换货、退款、投诉  | 生成退货标签、预审批退款 | 售后 API + 政策 RAG     |
| **合规解答** | 平台规则、保修、合规 | 严格基于文档       | 平台政策、物流规则           |
| **人工转接** | 复杂场景升级     | 排队转接         | escalate\_to\_human |

### 意图 -> Agent 映射

- `product_inquiry`, `price_discount`, `faq` → **咨询接待**
- `order_status`, `logistics_tracking`, `shipping_info` → **订单履约**
- `return_exchange`, `refund`, `complaint` → **售后处理**
- `policy_compliance`, `platform_rules`, `warranty` → **合规解答**
- `complex_escalation` → **人工转接**

## 三、RAG 检索系统

### 数据规模

- 产品手册、物流规则、平台政策、售后流程等 **10w+** 非结构化文档

### 处理流程

```
原始文档 → 清洗 → 语义分块 → 嵌入 → 向量索引
                ↓
           混合检索 (向量 + BM25) → 重排序 → Top-N 返回
```

### 技术选型

- **向量库**：ChromaDB
- **嵌入**：OpenAI text-embedding-3-small
- **混合检索**：向量相似度 + BM25 关键词
- **重排序**：按相关性分数取 Top-N，降低幻觉

## 五、多轮对话状态管理

### 问题

- 长会话信息丢失
- 意图偏移
- 上下文窗口超限

### 方案：结构化状态存储

```
messages: 仅保留最近 N 轮原始消息 (如 10 条)
turn_summaries: 更早轮次的 [intent, key_entities, summary]
user_context: 用户画像（订单数、偏好、历史问题类型）
current_intent / current_agent: 当前路由结果
```

### 压缩策略

- 超出 `MAX_RECENT_MESSAGES` 时，将较早消息转为 `TurnSummary` 写入 `turn_summaries`
- 传入 LLM 时：`summaries` + 最近 `messages`，避免全量上下文
